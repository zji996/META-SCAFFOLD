# Repository Patterns

仅在初始化、重组、文档治理、多服务编排或多实例环境任务中读取。模式是候选，不是默认模板。

## 结构选择

使用部署边界、团队所有权、依赖方向和验证成本判断目录边界：

- 独立运行、构建或部署的单元适合放 `apps/` 或 `services/`。
- 多个运行单元共享且不依赖具体 app 的能力适合放 `packages/` 或 `libs/`。
- 只有一个消费者且没有稳定复用价值的代码留在消费者内部。
- monorepo 适合需要统一原子修改、共享契约和统一验证的系统；独立发布、权限隔离或规模约束更强时保留 polyrepo。

不要仅因目录看起来整齐而搬家。边界变化应同时说明依赖、构建、测试和发布影响。

## 文档角色

推荐从最小集合开始，按真实需要增加：

- `AGENTS.md`：项目特有约束、命令和上下文入口。
- `docs/current.md`：近期焦点，建议不超过约五个下一步。
- `docs/reference/architecture.md`：当前架构事实。
- `docs/decision/00NN-*.md`：需要保留理由的方向决策；可加一页短 INDEX。
- `docs/roadmap.md`：未来方向和阶段目标。

第一次为既有项目记录已经存在的隐含决策，可以作为事实整理直接完成并提示 review；引入新的方向决策仍需项目约定的确认。

## 本地产物与执行账本

仓库需要持久但不入 git 的运行产物时，可统一放 `.local/` 并整体忽略。子目录沿用项目已有命名，例如 `run/`、`plan/`、`backlog/`，不要强制模板。

长目标可在 `.local/plan/plan.md` 顶部维护：当前焦点、下一个未完成项、阻塞和 checklist。稳定事实仍回到 current/reference/ADR。

## 多服务本地编排

多服务需要常驻时，优先在已有命令入口封装 build/start/stop/status/logs，并记录 pid 或进程句柄；避免让使用者手工维护一组裸后台命令。具体工具和目录服从仓库现状。

## 多实例端口

只有同一主机确实需要并行多套环境时，才引入 profile 级端口实例。一个可用模式是将实例前缀与默认端口的“首位 + 末两位”组合，例如实例 `12` 把 8080、5432、9000 映射为 12880、12532、12900。

实施时必须同时处理：

- 检测派生端口碰撞和 65535 上限。
- 保持容器内端口稳定，只偏移宿主暴露端口。
- 让检查脚本、文档和 curl 示例读取最终配置，而非硬编码旧端口。
- 提供 `ports` 类自省命令输出解析后的端口和角色。
- SSH 隧道两端使用同一实例规格；规格变化后重建隧道和长连接。
